NubBank BaaS — Card API Reference

Service baas-card · Phase 1C Track-Card · Card spine: products, issuance & lifecycle, per-card limits, BIN ranges

Overview

The Card API lets partners issue and manage card products and cards. All partner-facing endpoints are served under the base path /baas/v1.

Authentication

Every partner request must carry one of the following credentials. Both resolve to full tenant authority in Phase 1C (operator-JWT RBAC on card endpoints is deferred).

  • Partner JWT (HMAC-SHA256): Authorization: Bearer <jwt>
  • API key: Authorization: ApiKey <key>

Response envelope

Every response uses the standard envelope:

{
  "data":   { ... },     // resource payload (or null on error)
  "meta":   { ... },     // pagination / metadata
  "errors": [ ... ]      // array of { code, message } on failure
}
PAN safety: a card's full Primary Account Number (PAN) is never returned by any endpoint and is never logged. Card responses expose maskedPan only; the PAN is stored AES-GCM encrypted at rest.

Card Limits

Per-card spending limits. A card with no limit row set is treated as unlimited (the limits view returns all-null in that case).

Method & PathDescriptionRequest / Response
PUT
/baas/v1/cards/{id}/limits		 Set / update per-card limits. Returns 200. Request UpdateCardLimitsRequest (all optional): 
{
  "dailyPurchase": 1000000,
  "dailyWithdrawal": 500000,
  "perTxn": 200000,
  "monthly": 10000000
}
Response CardLimitResponse: 
{
  "cardId": "<uuid>", "dailyPurchase": 1000000,
  "dailyWithdrawal": 500000, "perTxn": 200000,
  "monthly": 10000000, "updatedAt": "<timestamp>"
}
GET
/baas/v1/cards/{id}/limits		 Read the current limits. Returns 200. Response: CardLimitResponse (all-null fields = unlimited).

BIN Ranges

Bank Identification Number ranges used to route card transactions to the owning partner. BIN values are normalized left-aligned and zero-padded to 8 characters.

Method & PathDescriptionRequest / Response
POST
/baas/v1/bins		 Register a BIN range. Returns 201. Request RegisterBinRangeRequest: 
{
  "binStart": "50677500",
  "binEnd": "50677599",
  "scheme": "VERVE"   // optional
}
Response BinRangeResponse: 
{
  "id": "<uuid>", "binStart": "50677500",
  "binEnd": "50677599", "scheme": "VERVE", "active": true
}
GET
/baas/v1/bins		 List registered BIN ranges. Returns 200. Response: array of BinRangeResponse (shape above).

Internal (service-to-service)

These endpoints are not partner-facing. They are authenticated with body-signed HMAC (InternalServiceAuthFilter) and carry no partner context. They are consumed by Track-FEP and both shapes are frozen in the cross-track interface contract.

Authorization decision (contract §2a)

Method & PathDescriptionRequest / Response
POST
/internal/v1/authorize		 Authorize a transaction for a card. Returns 200. Card resolves the card by pan_hash = HMAC-SHA256(app.encryption.key, pan). Request AuthorizationDecisionRequest: 
{
  "partnerId": "<uuid>",
  "schemaName": "partner_<uuid>",
  "pan": "<full PAN from DE2>",
  "amountMinor": 50000,        // long, minor units
  "currency": "566"            // ISO 4217 numeric
}
Response ApiResponse<AuthorizationDecisionResponse>, data: 
{
  "decision": "APPROVE",       // APPROVE | DECLINE
  "responseCode": "00",        // ISO 8583 DE39
  "message": "approved"
}

Response-code mapping (Phase 1C stub)

RC (DE39)Meaning
00Approve
56No such card
62Card blocked / cancelled
54Card not usable (ISSUED / EXPIRED)
61Exceeds per-transaction limit
The balance check is a Phase 1C stub — always sufficient (deferred: real balance check via baas-engine, DEF-1C-23).
PAN safety: the authorization decision stub logs only the resulting decision and response code — never the PAN, card id, or any cardholder data.
