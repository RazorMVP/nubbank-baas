NubBank BaaS — API Reference

Phase 1C · partner card APIs (baas-card, REST) and the ISO 8583 front-end processor (baas-fep, TCP)

baas-card — Card API

The Card API lets partners issue and manage card products and cards. All partner-facing endpoints are served under the base path /baas/v1.

Authentication

Every partner request must carry one of the following credentials. Both resolve to full tenant authority in Phase 1C (operator-JWT RBAC on card endpoints is deferred).

  • Partner JWT (HMAC-SHA256): Authorization: Bearer <jwt>
  • API key: Authorization: ApiKey <key>

Response envelope

Every response uses the standard envelope:

{
  "data":   { ... },     // resource payload (or null on error)
  "meta":   { ... },     // pagination / metadata
  "errors": [ ... ]      // array of { code, message } on failure
}
PAN safety: a card's full Primary Account Number (PAN) is never returned by any endpoint and is never logged. Card responses expose maskedPan only; the PAN is stored AES-GCM encrypted at rest.

Authorization decision (contract §2a)

Method & PathDescriptionRequest / Response
POST
/internal/v1/authorize		 Authorize a transaction for a card. Returns 200. Card resolves the card by pan_hash = HMAC-SHA256(app.encryption.key, pan). Request AuthorizationDecisionRequest: 
{
  "partnerId": "<uuid>",
  "schemaName": "partner_<uuid>",
  "pan": "<full PAN from DE2>",
  "amountMinor": 50000,        // long, minor units
  "currency": "566",           // ISO 4217 numeric (DE49)
  "stan": "000042",            // DE11 — 6-digit Systems Trace Audit Number
  "terminalId": "TERM0001",    // DE41 — 8-char terminal identifier
  "transmissionDateTime": "0604143022" // DE7 — MMDDhhmmss
}
Response ApiResponse<AuthorizationDecisionResponse>, data: 
{
  "decision": "APPROVE",       // APPROVE | DECLINE
  "responseCode": "00",        // ISO 8583 DE39
  "message": "approved"
}

Response-code mapping (Phase 1C)

RC (DE39)Meaning
00Approve
56No such card
62Card blocked / cancelled
54Card not usable (ISSUED / EXPIRED)
61Exceeds per-transaction limit (evaluated in the transaction currency, scaled by its ISO 4217 minor-unit exponent)
57Currency not permitted (limit currency mismatch)
12Unknown / unparseable currency code in request
Stage 5: the balance check is now real — an approved authorization debits the cardholder's engine account via POST /internal/v1/card-debit (DEF-1C-23 closed). Authorization idempotency is active: duplicate requests (same stan + terminalId + transmissionDateTime) return the cached decision, and the engine de-dupes the debit on the same key.
PAN safety: the authorization decision logs only the resulting decision and response code — never the PAN, card id, or any cardholder data.

Reversal decision (contract §2b)

Method & PathDescriptionRequest / Response
POST
/internal/v1/reversal		 Locate the original authorization and credit the cardholder account back via the engine (Stage 5). Returns 200. Internal HMAC-signed; not partner-facing. FEP maps located: true → DE39 00; located: false → DE39 25 (unable to locate / engine unreachable — terminal retries, credit is idempotent). Request ReversalDecisionRequest: 
{
  "partnerId": "<uuid>",
  "schemaName": "partner_<uuid>",
  "originalStan": "000042",           // DE90 original STAN
  "terminalId": "TERM0001",           // DE41 — 8-char terminal id
  "originalTransmissionDateTime": "0604143022" // DE90 original MMDDhhmmss
}
Response ApiResponse<ReversalDecisionResponse>, data: 
{
  "located": true   // true = original found and marked reversed
                    // false = no matching idempotency row (RC 25)
}
The FEP extracts originalStan and originalTransmissionDateTime from ISO 8583 DE90 (Original Data Elements) of the 0400 reversal message. Card matches against the per-tenant idempotency table keyed by stan|terminalId|transmissionDateTime, then credits the cardholder account via the engine (Stage 5 — DEF-1C-25 fund half closed).

Card→engine money (contract §2c — Stage 5)

Hosted on baas-engine behind the @Order(0) internal chain (InternalServiceAuthFilter). The card calls these to move real money. amount is major-unit; currency is ISO 4217 alphabetic (the card translates the DE49 numeric code). authKey = stan|terminalId|transmissionDateTime. The engine is the money-dedupe authority — debit+dedupe is one atomic transaction keyed by card_auth_debit.auth_key (UNIQUE).

Method & PathDescription
POST
/internal/v1/card-debit		 Single-message immediate debit. Body { partnerId, schemaName, accountId, authKey, amount, currency } → data: { outcome } where outcome ∈ DEBITED|INSUFFICIENT|ACCOUNT_INVALID|CURRENCY_MISMATCH. Card RC map: DEBITED→00 · INSUFFICIENT→51 · ACCOUNT_INVALID→78 · CURRENCY_MISMATCH→57; unreachable engine → 91 (fail closed). Idempotent on authKey.
POST
/internal/v1/card-credit		 Reversal credit. Body { partnerId, schemaName, authKey } → data: { located }. Credits the account back exactly once (idempotent); FEP maps located:true→00 / false→25.
POST
/internal/v1/account-lookup		 Issuance validation. Body { partnerId, schemaName, accountId } → data: { exists, active, currencyCode }. Card rejects issuance with LINKED_ACCOUNT_NOT_FOUND when exists=false.

Engine→card provisioning (contract §2d — Stage 5)

Method & PathDescription

Customers baas-engine · partner-scoped

Customer lifecycle management for a partner's end-customers. Every request is automatically scoped to the caller's partner schema. PII fields (BVN, NIN, phone, date of birth) are stored AES-GCM encrypted at rest via EncryptedStringConverter; blind-index columns enable equality/prefix search without decrypting every row. BVN and NIN are write-once at creation only and never returned in full (last 4 digits masked).

Identity verification deferred: Phase 1C captures BVN/NIN and displays them masked, but does not validate them against NIBSS/CBN. Real KYC-level progression driven by identity verification is deferred to Phase 2 (DEF-1C-30).

Accounts baas-engine · partner-scoped

Deposit-account lifecycle and money movement for a partner's end-customers. Every request is automatically scoped to the caller's partner schema. Account numbers are assigned from the partner's virtual-account range. A new account opens directly in ACTIVE status (there is no separate approval step). Monetary amounts are major-unit BigDecimal; currencyCode defaults to NGN when omitted. Every credit and debit writes one immutable Transaction row carrying the post-movement runningBalance.

Money gating by status: a FROZEN account still accepts credits (deposits) but rejects debits (withdrawals). A CLOSED account rejects both. The status gate is checked before any balance-floor check on a withdrawal.
Method & PathAuthDescriptionRequest / Response

Maker-checker (four-eyes): when this command is guarded — the partner has enabled ACCOUNT_OPEN in maker-checker config and the environment is PRODUCTION — the endpoint instead returns 202 ACCEPTED with a pending TaskResponse ({ id, status: "PENDING", commandType, madeBy, … }) and no account is created until a second operator approves the task. Partner RBAC — Users, Roles & API Keys baas-engine · partner-scoped

Granular access control for partners who need to give multiple operators restricted access to the BaaS console. Authority is now resolved per-request from DB role assignments via the V7 Flyway migration and the startup PartnerRbacReconciler — there is no longer a blanket "first-party gets everything" grant. Existing organisations were back-filled with PARTNER_ADMIN at reconciliation time (DEF-1C-15 resolved on branch feat/partner-rbac).

Permission codes

All partner endpoints carry a @PreAuthorize annotation that names the permission code required. Codes are evaluated by the per-request authority resolver against the caller's role assignments.

CodeGroupingWhat it gates
MANAGE_PARTNER_USERSadminCreate, list, get, deactivate/reactivate partner users; replace their role assignments.
MANAGE_ROLESadminCreate, list, update, delete custom roles; assign permissions to a role; list all available permissions.
MANAGE_API_KEYSadminIssue scoped API keys for the caller's organisation.

Built-in partner roles

Seeded by the V7 migration (tenant/V7__partner_rbac.sql). Built-in roles are marked built_in = true and cannot be edited or deleted (→ 409 BUILT_IN_ROLE).

Role nameSuperuser markerEffective permissionsNotes
PARTNER_ADMIN Yes All permissions (resolved dynamically — no stored grants) The authority resolver short-circuits on is_superuser = true and grants the full permission set without reading role_permissions. The reconciler back-fills existing users with this role on startup.
PARTNER_MAKER No READ_CUSTOMER, CREATE_CUSTOMER, UPDATE_CUSTOMER, READ_ACCOUNT, CREATE_ACCOUNT, DEPOSIT, WITHDRAW, READ_LOAN, CREATE_LOAN, INITIATE_PAYMENT, RUN_REPORT Operate and transact; cannot approve maker-checker items (approval gates deferred to Spec B).
PARTNER_APPROVER No READ_CUSTOMER, READ_ACCOUNT, READ_LOAN, RUN_REPORT (+ APPROVE_* when maker-checker ships — Spec B) Four-eyes checker; read-only until maker-checker migration adds approval grants.
PARTNER_VIEWER No READ_CUSTOMER, READ_ACCOUNT, READ_LOAN, RUN_REPORT Strictly read-only.

Maker-Checker (Four-Eyes) baas-engine · partner-scoped

A command-first maker-checker (four-eyes) framework that lets a partner require a second operator to approve a state-changing command before it executes (closes DEF-1C-13). Enforcement is per-partner, per-command, opt-in, and PRODUCTION-only: a command is guarded only when the partner has enabled it in maker-checker/config and the caller's environment is PRODUCTION. The first (and currently only) guarded command is ACCOUNT_OPEN — see the changed POST /baas/v1/accounts contract below. Every request is automatically scoped to the caller's partner schema, so a task id belonging to another organisation naturally returns 404 (enumeration-safe via tenant-schema isolation).

Command-first replay. When a guarded command is submitted, nothing is written to the domain tables — a PENDING task is recorded instead. On approval, the framework replays the identical service method the synchronous path uses (AccountService.open for ACCOUNT_OPEN) — never a stripped re-implementation. The replay + status change commit in one transaction; if the replay throws, the whole transaction rolls back and the task stays PENDING. Execute-exactly-once is enforced by a pessimistic row lock (findByIdForUpdate) + @Version optimistic guard + a status precondition.

baas-fep — ISO 8583 Front-End Processor TCP service

baas-fep is not a REST API. It is a stateless ISO 8583-1987 front-end processor: a raw TCP socket server that terminals (ATM / POS / scheme simulators) connect to. It holds no database and no tenant state — it parses each message, resolves the owning partner from the card BIN, forwards the authorization decision request to baas-card, and maps the result back onto the ISO 8583 response. The only HTTP surface is the actuator health endpoint used by orchestration probes.

BIN-based tenant routing

On 0100/0200, the FEP extracts the DE2 PAN and normalizes its leading digits to an 8-character BIN (take the first 8 digits, or fewer if the PAN is shorter; left-align and zero-pad to 8). It resolves the owning partner via baas-card GET /internal/v1/bins/{bin} → { partnerId, schemaName }, caching results in Caffeine for 5 minutes. The authorization request is then sent to baas-card POST /internal/v1/authorize with { partnerId, schemaName, pan, amountMinor, currency, stan, terminalId, transmissionDateTime }; the response { decision, responseCode, message } maps responseCode onto DE39. Both internal calls are signed with an inter-service HMAC and are consumed by contract (see docs/contracts/phase1c-interfaces.md §2 & §2a).

Unrouteable BIN → RC 91, no PAN echo. If the BIN resolves to no partner (or the lookup fails), the FEP returns DE39 91 and deliberately omits DE2 from the response — the PAN is never echoed on the unrouteable path. The PAN is never written to logs anywhere in the service.

EMV cryptogram validation, HSM, scheme-specific packagers/private DEs, settlement/advice MTIs, and tokenization are intentionally out of Phase 1C scope (deferred — see docs/deferred-items.md DEF-1C-01..07, plus FEP-specific DEF-1C-26). Stage 5 closed the money path: real balance-check debit (DEF-1C-23), reversal fund credit (DEF-1C-25), cross-service provisioning (DEF-1C-22), and the FEP authorization audit log (DEF-1C-24). Automatic GL double-entry posting remains out of scope (DEF-1C-27).
