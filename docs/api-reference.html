baas-fep — ISO 8583 Front-End Processor TCP service

baas-fep is not a REST API. It is a stateless ISO 8583-1987 front-end processor: a raw TCP socket server that terminals (ATM / POS / scheme simulators) connect to. It holds no database and no tenant state — it parses each message, resolves the owning partner from the card BIN, forwards the authorization decision request to baas-card, and maps the result back onto the ISO 8583 response. The only HTTP surface is the actuator health endpoint used by orchestration probes.

BIN-based tenant routing

On 0100/0200, the FEP extracts the DE2 PAN and normalizes its leading digits to an 8-character BIN (take the first 8 digits, or fewer if the PAN is shorter; left-align and zero-pad to 8). It resolves the owning partner via baas-card GET /internal/v1/bins/{bin} → { partnerId, schemaName }, caching results in Caffeine for 5 minutes. The authorization request is then sent to baas-card POST /internal/v1/authorize with { partnerId, schemaName, pan, amountMinor, currency }; the response { decision, responseCode, message } maps responseCode onto DE39. Both internal calls are signed with an inter-service HMAC and are consumed by contract (see docs/contracts/phase1c-interfaces.md §2 & §2a).

Unrouteable BIN → RC 91, no PAN echo. If the BIN resolves to no partner (or the lookup fails), the FEP returns DE39 91 and deliberately omits DE2 from the response — the PAN is never echoed on the unrouteable path. The PAN is never written to logs anywhere in the service.

EMV cryptogram validation, HSM, scheme-specific packagers/private DEs, settlement/advice MTIs, and tokenization are intentionally out of Phase 1C scope (deferred — see docs/deferred-items.md DEF-1C-01..07, plus FEP-specific DEF-1C-24..26).
